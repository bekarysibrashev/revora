## 1. Что такое Revora

AI-платформа управленческой аналитики для стоматологических клиник. Цель — дать собственнику/управляющему единую картину прибыли, денег, маркетинговой окупаемости и эффективности персонала вместо разрозненных 1С/банков/рекламных кабинетов.

**Первый (пилотный) клиент — SAN Dental Clinic**, 2 филиала: **SAN Seifullina** и **SAN Batysmura**. ТЗ на 6 листах Excel (`ТЗ_SAN_Dental_FINAL.xlsx`) описывает именно этого клиента, но продукт с самого начала строится как платформа для будущих клиник, а не одноразовое решение под SAN Dental.

**Важное обнаруженное расхождение (обсуждали в начале):** исходный питч Revora был про универсальную SaaS для "сотен клиник", а реальное ТЗ жёстко завязано на конкретные системы одного клиента (1С Стоматология, Kaspi/Halyk/Alatau, Facebook+2ГИС, Kcell). Решили: строим multi-tenant с первого дня, а специфику источников инкапсулируем в адаптерах, чтобы для второго клиента с другой МИС/банками не переписывать платформу.

## 3. Ключевые факты из ТЗ, которые нельзя забывать (мелочи)

- **Ядро данных — 1С Стоматология**, доступ через HTTP-сервис. Заказчик даёт тестовую базу для разработки.
- В карточке пациента 1С хранится **номер телефона и источник лида** — это основа сквозной аналитики (матчинг рекламы и продаж).
- **Выручка всегда выводится в двух видах одновременно**: по начислению и по оплате — это не один показатель с переключателем, а два разных факта.
- **Банки: только Kaspi, Halyk, Alatau**, только Excel, только ручная загрузка (банковских API нет).
- **Рекламные каналы: только Facebook (Meta) Ads и 2ГИС.** Google/Яндекс/VK были в обсуждении с заказчиком, но **явно удалены** из ТЗ — не добавлять их самопроизвольно.
- **2ГИС даёт рейтинг по конкретному врачу** (не только по клинике в целом).
- **Телефония — Kcell**, API уточняется у заказчика; обязательно проектировать с fallback на ручную выгрузку, чтобы не блокировать релиз.
- **GA4**: сайт у клиники есть, но GA4 к нему **не подключён** — это отдельная задача подключения, не просто интеграция с уже работающей аналитикой.
- **Доставка ежедневного отчёта — только Telegram-бот**, PDF, отправка в **08:00**.
- **Сегмент VIP-пациентов был в обсуждении и явно убран** из ТЗ по решению заказчика — не добавлять.
- **Управленческий учёт в 1С не настроен** на деление расходов на постоянные/переменные и на прямые/косвенные — это должна делать AI-классификация платформы, а не 1С.
- **4 роли, без гибкой RBAC-системы:**
  - Собственник/ГД — всё, все филиалы.
  - Управляющий — всё, включая ОПиУ и ДДС, **оба филиала с переключением**, но без управления ролями/настройками.
  - Администратор — только операционка/продажи, **свой филиал**, без финансов.
  - Менеджер по продажам — **только свои** сделки/лиды, ничего чужого.
- Комиссия/ЗП врача — смешанная схема, **25–40% от выручки ИЛИ фиксированная ставка** (см. решение #5 в таблице выше).
- Заказчик уже закрыл раунды вопросов сам с собой (лист «Вопросы заказчику» в исходном xlsx) — то есть базовые неопределённости в самом ТЗ уже сняты, наши уточнения — это уточнения архитектуры поверх готового ТЗ, а не переоткрытие вопросов заказчика.

## 4. Этапность внедрения (из ТЗ, легла в основу roadmap)

**MVP (1-я очередь по ТЗ заказчика):**
Интеграция с 1С (real-time HTTP-сервис) → Dashboard CEO (финансы+продажи+операционка) → Аналитика продаж и врачей → ОПиУ (деление пост./перем. вручную) → ДДС (загрузка выписок + полуручная категоризация) → Роли и доступ.

**v1 (2-я очередь по ТЗ):**
AI-категоризация выписок (авто-ДДС) → AI-классификация расходов ОПиУ → Сквозная аналитика Facebook↔1С (ROAS/ROMI по источнику) → Интеграции Meta Ads/2ГИС/Kcell/GA4 → Прогноз кассового разрыва и ликвидности.

**v2 (3-я очередь по ТЗ):**
AI-аналитик (чат по всем данным) → Ежедневный AI-отчёт в Telegram (08:00) → Прогнозирование показателей.

**v3 / Enterprise (наше расширение сверх ТЗ заказчика, для платформизации Revora):**
Онбординг новых клиник без разработчиков, новые адаптеры МИС/банков/рекламных каналов, бенчмаркинг между клиниками сети, мобильное приложение, white-label, SSO/SCIM, экспорт в DWH, SLA/on-prem, маркетплейс интеграций.

## 5. Архитектура — сводно (детали в `02_arhitektura_revora.md`)

- **Стек:** FastAPI (async) + NextJS + PostgreSQL + SQLAlchemy 2.0 (async) + Alembic + Redis + Celery + MinIO + JWT + Docker Compose.
- **Модульный монолит**, каждый модуль — вертикальный срез (`models / schemas / repository / service / router / dependencies`): `auth`, `tenancy`, `integrations`, `finance`, `sales`, `doctors`, `marketing`, `ai`, `dashboard`, `reports`, `billing` (скелет под будущий SaaS-биллинг), `admin`.
- **Слои:** API/Router (только HTTP) → Service (бизнес-правила, в т.ч. ролевые ограничения) → Repository (только доступ к БД). Core — инфраструктура (config, security, database, middleware). Shared — только реально переиспользуемое (деньги как Decimal, телефон в E.164, даты в Asia/Almaty).
- **Multi-tenancy:** `tenant_id` на каждой бизнес-таблице + Postgres RLS-политики; middleware ставит `SET LOCAL app.tenant_id/app.role/app.branch_ids/app.user_id` из claims JWT в начале транзакции. Тем же механизмом закрыт разрез по филиалам и по "своим" сделкам менеджера.
- **Integrations module = адаптеры.** `IntegrationAdapter`: `fetch → validate → normalize`. ETL: Extract (адаптер) → Transform (mapper в каноническую схему) → Load (факт-таблицы). Dashboard/AI никогда не видят специфику источника.
- **AI module:** `categorization/` (MVP — rules_engine по ключевым словам, v1 — ML/LLM-классификатор с человеческой обратной связью через `ai_classification_feedback`), `analyst/` (чат, LLM вызывается только через абстрактный `llm_provider.py`, никогда напрямую из других модулей), `daily_report/`.
- **Инфраструктура (docker-compose):** backend, worker, beat, postgres, redis, minio, frontend, nginx, flower (dev).

## 6. База данных — сводно (детали в `03_baza_dannyh_revora.md`)

- Транзакционное ядро — 3NF. Аналитические факт-таблицы (`revenue_facts`, `expense_facts`, `cash_flow_facts`, `marketing_spend_facts`) — намеренно денормализованная звезда для быстрого чтения дашбордов.
- Все деньги — `NUMERIC(14,2)` + отдельное поле `currency`, никогда `FLOAT`. Все времена — `TIMESTAMPTZ` в UTC, конвертация в `Asia/Almaty` только на отображении.
- ~25 таблиц по доменам: tenancy/auth (`tenants`, `branches`, `users`, `user_branches`, `refresh_tokens`), пациенты/лиды (`patients`, `leads`, `calls`), врачи (`doctors`, `doctor_compensation_rules`, `doctor_ratings`), записи (`service_directions`, `appointments`, `treatment_plans`), финансы (`revenue_facts`, `expense_categories`, `expense_facts`, `bank_statement_uploads`, `raw_bank_transactions`, `cash_flow_facts`, `account_balances`), маркетинг (`marketing_spend_facts`, `attribution_facts`), интеграции (`integration_connections`, `sync_runs`), AI/отчёты (`ai_classification_feedback`, `daily_reports`, `ai_chat_sessions`, `ai_chat_messages`), аудит (`audit_log`).
- **Идемпотентность** — UNIQUE-хэш на банковские транзакции (`transaction_hash`), UNIQUE `external_id` на записи из 1С — повторная загрузка/синк не создаёт дублей на уровне БД.
- **Телефон** хранится и зашифрованным (`phone_e164`), и как `phone_hash` (SHA-256) — матчинг рекламы и продаж идёт по хэшу, не по открытому номеру (PII-защита).
- **История ставок ФОТ** — `doctor_compensation_rules` с CHECK (ровно один из `percentage_value`/`fixed_amount`) + GIST exclusion constraint на непересечение периодов одного врача.

## 7. API — сводно (детали в `04_api_revora.md`)

- Базовый путь `/api/v1`, JWT Bearer, единый формат ошибок `{error: {code, message, details}}`, единая пагинация, единый паттерн фильтров `date_from/date_to/branch_id`.
- Каждый ответ аналитических модулей содержит `meta.data_as_of` — честная метка свежести данных (важно, т.к. 1С синкается раз в 1–3 часа, а не в реальном времени).
- Модули эндпоинтов: `auth`, `admin` (филиалы/пользователи/интеграции/аудит), `finance` (ОПиУ/ДДС/расходы/загрузка выписок — асинхронная, `202`), `dashboard` (CEO + операционка), `sales` (обзор/воронка/план-факт/лиды), `doctors` (карточки/рейтинги/ЗП-правила/выплаты), `marketing` (обзор/кампании/атрибуция), `ai` (чат-сессии, ежедневные отчёты).
- Матрица доступа по ролям задокументирована по каждому модулю; реализуется двойным контролем — `Depends(require_role(...))` в роутере + RLS в БД.
- `POST /doctors/{id}/compensation-rules` — при создании новой схемы автоматически закрывает предыдущую (`valid_to`), конфликт периодов → `409 OVERLAPPING_COMPENSATION_PERIOD`.

  каноническую структуру; ошибочные строки формируют карантинные причины. Следующий срез —
  persistence/orchestration и первый реальный адаптер, после него — витрины и доменные API.
