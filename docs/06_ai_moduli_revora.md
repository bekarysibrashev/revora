## 0. Решение по объёму этапа (закрывает вопрос из раздела 8 предыдущего документа)

**Решение №11:** AI-чат и ежедневный отчёт остаются заглушкой «СКОРО» (v2), но **вся аналитика — поиск потерь, анализ врачей/админов/маркетинга, прогнозирование, обнаружение аномалий — проектируется и реализуется сейчас**, как **фоновые бэкенд-джобы**, результат которых — **алерты/инсайты на дашборде**, а не текст в чате или PDF-отчёте. Когда в v2 появится чат и ежедневный отчёт, они станут ещё одним *способом подачи* уже готовых данных из этого слоя (тот же источник, другая упаковка), а не отдельной аналитической логикой.

Ключевой архитектурный принцип, который отсюда следует и держит всё остальное в этом документе:

> **Цифры всегда считает детерминированный код (SQL/rules-engine), LLM никогда не вычисляет и не "придумывает" финансовые значения.** LLM используется только там, где нужна классификация/разметка (категория расхода, статья ДДС) или — в будущем, в v2 — упаковка уже посчитанных цифр в естественный язык для чата/отчёта. Это защищает продукт управленческой аналитики от галлюцинаций в самом чувствительном месте — деньгах.

## 1. Общая архитектура AI-модуля (уточнение раздела 5 из `02_arhitektura_revora.md`)

```
ai/
├── categorization/        # MVP: rules_engine, v1: ML/LLM-классификатор + human feedback
│   ├── rules_engine.py
│   ├── llm_classifier.py          # v1, через llm_provider.py
│   └── feedback_service.py        # запись/чтение ai_classification_feedback
├── insights/               # НОВОЕ на этом этапе — вся аналитика ниже
│   ├── loss_detection/            # поиск потерь
│   ├── performance_analysis/      # врачи / админы / маркетинг
│   ├── forecasting/               # кассовый разрыв (v1) → метрики (v2)
│   ├── anomaly_detection/         # статистические аномалии и риски
│   └── insight_repository.py      # запись в ai_insights, общий для всех джобов выше
├── analyst/                # v2 — стаб, не реализуется в Этапе 7
├── daily_report/           # v2 — стаб, не реализуется в Этапе 7
└── llm_provider.py         # единственная точка входа к LLM для всего модуля ai/
```

**Почему `insights/` — это не `analyst/`:** `analyst/` в изначальной архитектуре был спроектирован как модуль вокруг чата (сессии, сообщения, LLM-диалог). Аналитика, которую мы делаем сейчас, не диалоговая — это регулярно пересчитываемые правила и статистика, результат которых складывается в таблицу и показывается значками/списком на дашборде. Это разные модули с разным жизненным циклом, поэтому разносим их, а не тащим бизнес-логику внутрь будущего чата.

**Cadence джобов** — наследует уже принятое решение №7 (раздел 2 мастер-контекста) по частоте синхронизации источников: джоб не может пересчитывать инсайт чаще, чем обновляются данные, на которых он основан.

| Джоб | Частота | Почему |
|---|---|---|
| Категоризация новых банковских транзакций | по факту загрузки выписки (событийно) | выписка загружается вручную, разово |
| Категоризация расходов ОПиУ | синхронно с синком 1С (каждые 1–3 часа, Celery beat) | новых расходов из 1С не больше, чем раз в 1–3 часа |
| Сквозная аналитика ROAS/ROMI | каждые 1–3 часа (лимитируется 1С-стороной матчинга) | реклама обновляется в реальном времени, но матчинг с продажами упирается в частоту синка 1С |
| Поиск потерь | 1 раз в час | не нужно чаще операционных данных, но и не должно быть редким — часть потерь (лид без ответа) теряет ценность при задержке в сутки |
| Анализ врачей/админов/маркетинга | 1 раз в сутки (ночью) | это трендовая, не оперативная аналитика |
| Прогноз кассового разрыва | 1 раз в сутки | входные данные — ДДС, обновляется не чаще загрузки выписок |
| Обнаружение аномалий | 1 раз в час по операционным метрикам, 1 раз в сутки по финансовым | баланс между "не спамить алертами" и "не проспать риск" |

## 2. Категоризация выписок (авто-ДДС) — MVP → v1

**MVP (rules_engine):**
- Правила матчатся по: назначению платежа (ключевые слова/regex), контрагенту (ИНН/название), диапазону сумм — в таком порядке приоритета (более специфичное правило побеждает).
- Таксономия статей — уже существующая `expense_categories` (раздел 6 мастер-контекста), правила не изобретают новые категории, только сопоставляют транзакцию существующей.
- Результат матчинга — `confidence` (высокий/средний/низкий по числу совпавших признаков). Ниже порога — транзакция уходит в очередь **ручной категоризации**, не проставляется автоматически (не додумываем за пользователя там, где неуверенно).
- Ручная категоризация пользователем **обязательно** пишется в `ai_classification_feedback` — это топливо для v1.

**v1 (ML/LLM-классификатор):**
- Классификатор дообучается/использует few-shot на основе накопленного `ai_classification_feedback` конкретного тенанта (у разных клиник разные контрагенты и формулировки платежей — модель не должна быть общей для всех).
- LLM вызывается **только** через `llm_provider.py`, получает на вход текст назначения платежа + контрагента + сумму + список допустимых категорий (закрытый список, не свободная генерация) → возвращает одну категорию из списка + confidence. Замкнутый список категорий — намеренное ограничение, чтобы LLM не мог "изобрести" несуществующую статью.
- Порог confidence для авто-простановки в v1 может быть ниже, чем в MVP (модель точнее rules_engine), но ручная очередь остаётся всегда — полностью убрать её нельзя (это финансовые данные).

Идентичная логика — для **AI-классификации расходов ОПиУ** (постоянные/переменные, прямые/косвенные): те же rules_engine → LLM-classifier → feedback loop, только категории другие (не статья ДДС, а тип расхода) и источник данных — не банковская выписка, а расходы из 1С.

## 3. Сквозная аналитика Facebook ↔ 1С (ROAS/ROMI)

- Матчинг — по `phone_hash` (раздел 6 мастер-контекста: матчинг рекламы и продаж идёт по хэшу телефона, не по открытому номеру) + `источник лида` из карточки пациента 1С.
- Результат матчинга пишется в `attribution_facts`: лид → источник (Facebook-кампания/2ГИС) → сделка (если состоялась) → выручка.
- Расчёт на уровне источника/кампании за период:
  - **ROAS** = выручка от сделок с этим источником / затраты на этот источник.
  - **ROMI** = (выручка − затраты) / затраты.
- Оба показателя считаются и **по начислению, и по оплате** — то же правило "выручка в двух видах", что зафиксировано в разделе 3 мастер-контекста, распространяется и на маркетинговую аналитику, не только на общий дашборд.
- 2ГИС даёт рейтинг по врачу, но не рекламный бюджет с точностью до заявки в том же смысле, что Meta Ads API — для 2ГИС ROAS/ROMI считается на уровне канала в целом, не кампании (ограничение источника данных, не архитектуры).

## 4. Поиск потерь (loss detection)

Конкретный список правил MVP-уровня (детерминированные пороги, настраиваемые в конфиге тенанта, не хардкод):

| # | Тип потери | Правило | Источник данных |
|---|---|---|---|
| 1 | Лид без реакции | Лид создан, но нет ни звонка, ни записи через N часов (по умолчанию 24) | `leads`, `calls`, `appointments` |
| 2 | Незакрытая сделка без причины | Запись/план лечения создан, но статус не менялся дольше N дней | `appointments`, `treatment_plans` |
| 3 | Невыполненный план лечения | `treatment_plans` начат, но не завершён, и с последней активности прошло больше N дней | `treatment_plans` |
| 4 | Разрыв начисление/оплата | Выручка начислена, но не оплачена дольше обычного срока клиники (медиана по тенанту + допуск) | `revenue_facts` (начисление vs оплата) |
| 5 | Дорогой канал | CAC (стоимость лида/продажи) по источнику выше среднего по клинике более чем на X% | `attribution_facts`, `marketing_spend_facts` |

**Открытый момент, который не выдумываю, а фиксирую как есть:** правило «неявка на приём» (no-show) — классический источник потерь для клиники, но в разделе 6 мастер-контекста явного поля статуса записи (`completed`/`no_show`/`cancelled`) в описании таблицы `appointments` нет — это может быть, а может не быть в реальном ответе HTTP-сервиса 1С. **Нужно подтвердить у заказчика при работе с тестовой базой на Этапе 7**, есть ли такой статус в 1С. Если есть — добавляем правило №6 «no-show» без изменения архитектуры (это ещё одно правило в том же движке). Если нет в 1С в чистом виде — выводим no-show косвенно (запись есть, оплаты/выполнения нет, дата в прошлом) с более низким confidence.

## 5. Анализ врачей / администраторов / маркетинга

Не диалоговая аналитика, а регулярно пересчитываемые метрики, каждая из которых может выродиться в алерт при выходе за порог (связь с разделом 8 — аномалии).

**Врачи:**
- Выручка по врачу (начисление и оплата) за период, план-факт (если план задан).
- Загрузка (факт записей / вместимость расписания) — источник: отчёт по загрузке кабинетов из 1С (раздел «Вопросы заказчику» ТЗ, п.12).
- Рейтинг 2ГИС по врачу в динамике.
- ФОТ врача за период — расчёт через `doctor_compensation_rules` (учитывает историю ставок, реальный тип начисления на каждый день периода, а не текущий).

**Администраторы:**
- Конверсия лид → запись → сделка по администратору, свой филиал (согласуется с ролевой моделью: админ видит только операционку своего филиала — эта же метрика видна ему в его собственном разрезе, Собственнику/Управляющему — по всем администраторам).

**Маркетинг:**
- ROAS/ROMI по каналу и кампании (раздел 3 этого документа) в динамике, не только моментальный срез — важен тренд, а не только текущее число.
- Стоимость лида по каналу в динамике.

## 6. Прогнозирование

**v1 — прогноз кассового разрыва и ликвидности** (единственный вид прогноза, явно заявленный в ТЗ и роадмапе как v1, раздел 4 мастер-контекста):
- Метод для старта — не ML, а прозрачная модель: текущий остаток по счетам (`account_balances`) + запланированные/типичные поступления и расходы (медиана по историческим `cash_flow_facts` за сопоставимый период) → проекция остатка на N дней вперёд, с алертом, если проекция уходит в минус раньше порогового горизонта (например, 14 дней).
- Осознанно **не ML-модель на старте**: для управленческого решения "хватит ли денег на зарплату через 2 недели" объяснимая арифметика надёжнее чёрного ящика, и её проще предъявить собственнику как обоснование алерта.

**v2 — прогнозирование прочих показателей** (выручка, поток лидов и т.д.) — это отдельный, явно более поздний по роадмапу пункт (раздел 4 мастер-контекста относит "Прогнозирование показателей" к v2, в отличие от кассового разрыва). Инфраструктура джобов (`forecasting/`) проектируется расширяемой под это уже сейчас (единый интерфейс "джоб на вход — исторический ряд, на выход — прогноз + доверительный интервал"), но конкретные модели для выручки/лидов не реализуются в Этапе 7 — только кассовый разрыв.

## 7. Обнаружение аномалий и рисков

Разделяю на два класса — намеренно разными методами, чтобы не переусложнять там, где не нужно:

**Статистические аномалии по операционным/финансовым метрикам** (выручка, расходы, поток лидов, маркетинговые затраты): отклонение текущего значения от скользящего среднего за сопоставимый период больше, чем на K стандартных отклонений (простой и объяснимый метод — z-score по скользящему окну, не ML). Порог K настраивается на тенант, дефолт — обсуждается эмпирически на реальных данных SAN Dental после накопления истории (в первые недели работы платформы аномалии по статистике попросту не считаются — недостаточно исторического окна, это ограничение метода, а не баг).

**Операционные риски, не про "необычное число", а про "надвигающееся событие":**
- Истекающая схема ФОТ врача без новой (пробел в `doctor_compensation_rules`).
- Повторяющиеся сбои синхронизации с источником (`sync_runs` — несколько неудачных подряд) — это уже про здоровье самой платформы, не про бизнес клиники, но напрямую влияет на доверие к цифрам на дашборде (см. `meta.data_as_of`), поэтому логично живёт в том же алертинге.
- GA4/Kcell без данных дольше ожидаемого интервала синка — аналогично, риск "тишина в источнике", не различимый на глаз от "у клиники правда не было звонков".

## 8. Доставка: модель данных и API для алертов/инсайтов

Расширение БД и API сверх того, что уже зафиксировано в `03_baza_dannyh_revora.md` / `04_api_revora.md` — **нужно перенести туда** при следующем обновлении этих документов.

**Новая таблица `ai_insights`:**
`id, tenant_id, branch_id (nullable — null = про клинику в целом), type (enum: loss/performance/forecast/anomaly/risk), severity (info/warning/critical), title, message, entity_type, entity_id, metric_value, threshold_value, data_as_of, created_at`.

**Новая таблица `ai_insight_reads`** (алерт общий на тенант/филиал, но прочитанность — персональная): `insight_id, user_id, read_at, dismissed_at`.

**Новые эндпоинты:**
- `GET /api/v1/dashboard/insights?severity=&type=&branch_id=` — список, отфильтрованный той же ролевой моделью, что и остальной дашборд (RLS + `require_role`).
- `POST /api/v1/dashboard/insights/{id}/dismiss` — персональное скрытие (пишет в `ai_insight_reads`, не удаляет сам инсайт — он общий для тенанта).

**На фронте** (дополнение к `05_frontend_revora.md`): значок-колокольчик в шапке (как в Linear/Notion) со счётчиком непрочитанных, список инсайтов группируется по `severity`, критические (`critical`) дублируются как баннер на самом дашборде, а не только в выпадающем списке — потерю или риск не должно быть можно пропустить, провалившись сразу в конкретный аналитический раздел.
