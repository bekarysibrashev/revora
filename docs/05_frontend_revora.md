## 1. Стек и версии

| Слой | Технология |
|---|---|
| Фреймворк | Next.js (App Router), TypeScript strict |
| Стили | Tailwind CSS |
| UI-кит | shadcn/ui (Radix primitives) |
| Серверное состояние / кэш | TanStack React Query |
| Формы | React Hook Form + Zod (схемы валидации переиспользуются между формой и типами API-ответов) |
| Графики | Recharts |
| Анимации | Framer Motion — точечно (переходы, микро-интеракции), не как основа UI |
| HTTP-клиент | тонкая обёртка над `fetch` с автоподстановкой `Authorization`, обработкой единого формата ошибок `{error:{code,message,details}}` и авто-рефрешем токена |
| Даты/локаль | `date-fns` + `Asia/Almaty` как таймзона отображения по умолчанию (данные в БД — UTC, конвертация только на фронте) |

**Не берём:** Redux/Zustand как основной стор — серверное состояние полностью на React Query, клиентского глобального состояния (UI-стейт: открыт ли сайдбар, выбранный филиал в сессии) достаточно через React Context + `nuqs`/URL-state там, где состояние должно жить в адресной строке (фильтры, даты, выбранный филиал — чтобы дашборд был шарабельной ссылкой).

## 2. Структура проекта

Вертикальные срезы по модулям — то же самое архитектурное решение, что и в backend (раздел 5 мастер-контекста), чтобы модуль на фронте 1:1 отражал модуль API.

```
frontend/
├── app/                          # Next.js App Router — только роутинг и layout-композиция
│   ├── (auth)/
│   │   └── login/
│   ├── (app)/                    # всё, что требует авторизации, общий layout с сайдбаром
│   │   ├── dashboard/
│   │   ├── finance/
│   │   │   ├── pnl/              # ОПиУ
│   │   │   └── cashflow/         # ДДС + загрузка выписок
│   │   ├── sales/
│   │   ├── doctors/
│   │   ├── marketing/
│   │   ├── reports/
│   │   ├── ai/                   # заглушка «СКОРО»
│   │   └── admin/                # филиалы/пользователи/интеграции/аудит — видно не всем ролям
│   └── layout.tsx
├── modules/                      # бизнес-логика по срезам, app/ только импортирует
│   ├── auth/       {api, hooks, components, schemas}
│   ├── dashboard/  {...}
│   ├── finance/    {...}
│   ├── sales/      {...}
│   ├── doctors/    {...}
│   ├── marketing/  {...}
│   ├── reports/    {...}
│   ├── ai/         {...}         # только Stub-компонент + запись в нав. меню
│   └── admin/      {...}
├── shared/
│   ├── ui/                       # обёртки над shadcn-компонентами, кастомизированные под дизайн-токены
│   ├── api-client/                # fetch-обёртка, обработка ошибок, refresh-token
│   ├── hooks/                     # useCurrentBranch, useRole, useDataFreshness и т.д.
│   ├── lib/                       # money-formatting (Decimal→KZT), phone, date-tz
│   └── types/                     # типы, сгенерированные/согласованные с API-контрактами
├── styles/
│   └── globals.css                # design tokens как CSS-переменные
└── middleware.ts                  # проверка JWT, редирект на /login, ролевые ограничения на роуты admin/*
```

Правило по модулю (зеркалит backend): **`api/`** — только вызовы эндпоинтов и React Query хуки (`useFinancePnl`, `useUploadBankStatement`…), **`components/`** — презентационные и контейнерные компоненты модуля, **`schemas/`** — Zod-схемы форм и рантайм-валидация ответов API. Ничего из внутреннего устройства модуля не импортируется напрямую другим модулем — только через его публичный `index.ts`.

## 3. Роутинг ↔ роли ↔ модули API

Прямое отражение матрицы доступа из раздела 7 мастер-контекста (`04_api_revora.md`) на фронт. Два уровня защиты, как и на бэкенде: **middleware** (грубая фильтрация — весь раздел недоступен роли) + **компонентный уровень** (тонкая фильтрация — часть виджетов/колонок скрыта, но раздел открыт).

| Раздел | Собственник/ГД | Управляющий | Администратор | Менеджер продаж |
|---|---|---|---|---|
| `/dashboard` | всё, оба филиала | всё, оба филиала (переключение) | операционка, свой филиал | нет |
| `/finance/*` (ОПиУ, ДДС) | ✅ | ✅ | ❌ | ❌ |
| `/sales` | ✅ | ✅ | операционка/продажи, свой филиал | только свои лиды/сделки |
| `/doctors` | ✅ | ✅ | просмотр, свой филиал | ❌ |
| `/marketing` | ✅ | ✅ | ❌ | ❌ |
| `/reports` | ✅ | ✅ | ❌ | ❌ |
| `/ai` (заглушка) | ✅ | ✅ | ❌ | ❌ |
| `/admin` | ✅ (роли/настройки) | без управления ролями | ❌ | ❌ |

Переключатель филиала в шапке виден только ролям, у которых `branch_ids` содержит больше одного филиала (Собственник, Управляющий); Администратор и Менеджер видят филиал как статичную метку, не переключатель — сервер всё равно режет доступ через RLS, но UI не должен показывать несуществующую свободу выбора.

## 4. Дизайн-система

Ориентир — Stripe / Linear / Notion: спокойный, плотный по данным, но не тесный интерфейс для управленческой аналитики, светлая тема, минимализм. Ниже — компактный токен-план (по методике, а не «дефолтная» цветовая пара).

**Цвет** (описываю как именованные значения, финальные hex уточняются при верстке первого экрана):
- `bg-canvas` — почти белый, чуть тёплый серый (не чистый `#FFFFFF`, чтобы карточки на нём читались без теней-«подушек»)
- `bg-surface` — белый, для карточек/таблиц
- `border-subtle` — очень светлый серый, hairline-разделители вместо теней там, где это уместно (в духе Linear)
- `text-primary` / `text-secondary` — почти чёрный / серый средней плотности
- `accent-brand` — один акцентный цвет Revora (не терракотовый «Claude-тон» и не кислотно-зелёный — сознательно уходим от AI-дефолтов), используется точечно: активные состояния, ключевые CTA, положительная динамика
- `signal-negative` / `signal-positive` — для денежных дельт (ОПиУ, ДДС), т.к. это финансовый продукт, где красный/зелёный несут смысл, а не декорацию

**Типографика:** один нейтральный гротеск для UI и данных (таблицы, суммы — моноширинные цифры/tabular-nums обязательно, чтобы колонки чисел в ОПиУ/ДДС визуально выравнивались), без отдельного «дисплейного» шрифта — это рабочий инструмент для собственника клиники, а не маркетинговая страница, вычурная типографика здесь будет мешать, а не помогать.

**Слой/плотность:** таблично-дашбордный UI живёт на границе высокой плотности данных и читаемости — берём вертикальный ритм и отступы уровня Linear (плотнее, чем типичный SaaS-дашборд, но не Bloomberg-terminal).

**Сигнатурный элемент раздела:** индикатор свежести данных (`meta.data_as_of` из API, раздел 7 мастер-контекста) — не техническая мелочь, а честный, всегда видимый элемент интерфейса (например, скромная метка у заголовка каждого аналитического блока: «данные на 14:32»), т.к. 1С синкается раз в 1–3 часа и владельцу важно не путать это с реал-таймом.

**Motion:** сдержанно — переход между филиалами/датами в дашборде, раскрытие строк в ОПиУ/ДДС, состояние загрузки при обработке банковской выписки (async `202` → polling, см. п.6). Никакой decorative-анимации на посадочных элементах — это не лендинг.

## 5. Паттерны данных и состояний

- **React Query** — один хук на один эндпоинт-юзкейс (`useDashboardCeo(branchId, dateFrom, dateTo)`), ключи кэша включают `tenant_id` неявно (он в JWT/контексте запроса), явно — фильтры. `staleTime` для аналитических эндпоинтов настраивается с оглядкой на `meta.data_as_of`, а не наугад.
- **Фильтры в URL** (`date_from`, `date_to`, `branch_id`) — единый паттерн из API (раздел 7) зеркалится в query-параметры фронта через `nuqs`, чтобы вкладка/ссылка на конкретный срез дашборда были шарабельны и переживали reload.
- **Формы** — RHF + Zod, схема формы по возможности выводится из той же Zod-схемы, что валидирует ответ API (единый источник правды на форму + рантайм-проверку), самый показательный пример — форма `POST /doctors/{id}/compensation-rules` с клиентской проверкой «ровно один из процента/фиксированной ставки» до отправки, дублируя серверный `CHECK`, чтобы не тратить `409 OVERLAPPING_COMPENSATION_PERIOD` на банальные ошибки ввода.
- **Единый формат ошибок API** (`{error:{code,message,details}}`) разбирается один раз в `api-client`, из `code` строится маппинг на человекочитаемое сообщение (справочник кодов на фронте, не на бэке).

## 6. Асинхронная загрузка банковских выписок (UX)

Отражение решения №6 из мастер-контекста (async `202 Accepted` + `upload_id`, отдельный polling-эндпоинт статуса):

1. Пользователь загружает Excel-выписку (Kaspi/Halyk/Alatau) → фронт получает `upload_id`, сразу показывает строку со статусом «Обрабатывается» в списке загрузок ДДС.
2. React Query с `refetchInterval` опрашивает статус по `upload_id`, пока не получит терминальный статус (`done`/`failed`).
3. По завершении — уведомление (toast) + автообновление списка `raw_bank_transactions`/`cash_flow_facts` без перезагрузки страницы.
4. Ошибка обработки показывается с деталями из `error.details` (например, нераспознанные строки выписки), а не общим «что-то пошло не так».

## 8. Экспорт в Excel/PDF (решение №10) — детали для API и UI

Вопрос №2 из раздела 8 мастер-контекста закрыт: **экспорт нужен в MVP**, не переносится в v1. Это меняет `04_api_revora.md` — там нужно завести новый эндпоинт-паттерн, которого пока нет. Фиксирую здесь, чтобы не забыть перенести при следующем обновлении API-документа:

- **Где нужна кнопка «Экспорт»**: `finance/pnl` (ОПиУ), `finance/cashflow` (ДДС), список расходов внутри ОПиУ/ДДС. Кнопка стоит рядом с фильтрами (`date_from/date_to/branch_id`) — экспортируется именно тот срез, что сейчас на экране, а не «всё подряд».
- **Формат эндпоинта** (по аналогии с уже принятым паттерном асинхронной загрузки выписок, раздел 6 этого документа, но здесь синхронно проще — генерация отчёта в разумных пределах не такая тяжёлая операция, как разбор банковской выписки):
  - `GET /api/v1/finance/pnl/export?format=xlsx&date_from=...&date_to=...&branch_id=...` и аналогично `GET /api/v1/finance/cashflow/export`.
  - `format` — `xlsx` или `pdf`, оба варианта нужны (PDF — для передачи «как есть» руководителю/в бухгалтерию, Excel — для дальнейшей работы бухгалтера с цифрами).
  - Ответ — бинарный файл с `Content-Disposition: attachment`, либо (если генерация не мгновенная) тот же паттерн `202 + upload_id`-подобный `job_id` с поллингом, что и для банковских выписок — решим на Этапе 6/7, когда будет понятна реальная скорость генерации Excel/PDF с реальным объёмом данных клиники.
- **Права доступа**: те же роли, что видят сам раздел (Собственник, Управляющий) — Администратор и Менеджер продаж кнопку экспорта не видят, т.к. не видят и сами ОПиУ/ДДС (раздел 3 этого документа).
- **На фронте**: отдельный хук `useExportReport(module, filters, format)` в `modules/finance/api/`, без разделяемого состояния — это разовое действие (клик → скачивание файла или переход в polling-состояние кнопки), в общий React Query кэш аналитики не пишет.
